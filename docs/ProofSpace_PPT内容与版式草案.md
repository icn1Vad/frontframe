## 1. 当前结论

ProofSpace 是面向企业文件审校、归档和知识利用场景的数据治理产品。产品前端把文件上传、AI 分类、人工确认、分类任务、审查任务、风险报告、正式入库、知识检索和智能问答组织为一条可追溯流程。

项目当前包含两个前端：

| 项目 | 作用 | 当前技术 |
| --- | --- | --- |
| `newframe` | 产品展示站，负责定位、产品流程、技术架构和推进路线 | Next.js 16、React 19、TypeScript、Tailwind CSS、Three.js / React Three Fiber、Framer Motion |
| `frontframe` | 业务管理端，承载文件治理实际操作界面 | Next.js 15 Pages Router、React 19、TypeScript、Lucide React、全局 CSS |

### 事实边界

- **已实现（前端可见）**：工作台、文件分类、分类任务、审查任务、审查报告、知识库、智能问答，以及展示站的产品和架构页面。
- **已有明确工程方案**：ReactAgent、TaskManager、Scheduler、Registry、Data Access Gateway、SQLite/Redis 双层上下文。
- **需要代码或运行数据进一步证明**：完整 Agent Harness 后端、端到端数据接入、权限隔离、审计追踪、上下文压缩效果和性能指标。
- **不能在 PPT 中直接声称**：市场规模、客户数量、准确率、效率提升比例、营收、融资和商业化成果，除非补充真实来源。

## 2. 建议的核心叙事

### 一句话定位

**ProofSpace 是一个以 Agent Harness 为技术底座、覆盖企业文件“分类—审查—归档—问答”全流程的智能数据治理平台。**

### 核心问题

企业文件治理通常被拆散在文件系统、人工审批、审查工具和知识库中，导致：

1. 文件分类和审查依赖人工经验，标准不统一。
2. AI 处理过程不可追溯，结果难以直接进入正式业务流程。
3. 审查结果与知识库脱节，同类问题需要重复判断。
4. Agent 若直接访问数据源，权限、审计和数据隔离难以治理。

### 产品解法

将 AI 能力放进受控工作流，而不是只提供一个聊天入口：

`上传文件 → AI 识别与分类 → 人工确认 → 分类任务 → 审查任务 → 风险报告 → 正式归档 → 知识库 → 可追溯问答`

**标题**：企业文件治理不缺工具，缺少贯通流程

**核心结论**：文件分类、内容审查、归档和知识问答彼此割裂，AI 结果难以成为可信的企业知识资产。

### 第 5 页：产品流程

**标题**：一条流程完成文件从输入到知识复用

**流程**：

1. 上传文件并识别类型、层级和推荐分类。
2. 人工确认后生成分类任务。
3. 进入内容审查，识别语义风险和条款冲突。
4. 输出结构化审查报告并处理风险。
5. 审查通过后正式入库。
6. 基于正式知识库进行可追溯问答。

### 第 10 页：技术创新一

**标题**：三层架构把 Agent 能力放进可治理边界

**三层结构**：

1. **Single Agent Runtime**：ReactAgent、提示词和消息管理、ReAct 循环、工具调用、Token 压缩。
2. **Agent Harness Control**：TaskManager、Scheduler、Registry、Session、Trace、SSE、权限和并发控制。
3. **Data Governance**：所有 SQL、RAG、文件、知识库和外部 API 访问统一经过 Data Access Gateway。

**创新点**：Agent 不直接连接数据库，数据访问在统一入口完成鉴权、隔离和审计。

**视觉**：三层架构图，数据网关横向贯穿底层。

---

### 第 11 页：技术创新二

**标题**：TaskManager + Scheduler 将模型调用升级为任务系统

**核心结论**：任务管理和调度层把一次模型响应扩展为可恢复、可追踪、可并发控制的业务任务。

**模块职责**：

- TaskManager：任务创建、状态流转、取消和重试。
- Scheduler：任务分发、并发控制和执行顺序。
- Registry：Agent 与工具的注册、发现和版本管理。
- Session / Trace：会话状态、运行链路和问题定位。

**视觉**：请求进入任务管理器，经调度器选择 Agent，再调用工具并返回事件流。

---

### 第 12 页：技术创新三

**标题**：双层上下文与统一网关兼顾连续性和数据安全

**上下文机制**：

- SQLite 保存完整会话历史，支持恢复和审计。
- Redis 保存运行态上下文，支持低延迟读写。
- 超出预算时优先压缩，压缩失败再裁剪兜底。

**数据治理机制**：

- Tool / Data Gateway 统一访问入口。
- 用户权限、租户隔离和调用审计在网关执行。
- SQL、RAG 和文件接口不暴露给 Agent 直接调用。

**视觉**：左侧双层记忆，右侧数据网关，中央为 Agent Harness。

## 5. PPT 版式规范

### 基础规格

- 页面比例：16:9。
- 建议尺寸：1920 × 1080 设计基准。
- 背景：暖白 `#F7F2EA`。
- 主文字：近黑 `#1E1B18`。
- 次级文字：灰褐 `#6E6861`。
- 强调色：铜橙 `#C57832`。
- 深色结构块：炭灰 `#393A35`。
- 状态色只用于语义：绿色表示已完成，黄色表示待复核，红色表示高风险。

### 字体层级

- 封面标题：36–44 pt。
- 页面标题：28–32 pt。
- 核心结论：18–22 pt。
- 正文：14–17 pt。
- 图注和导航：9–11 pt。
- 中文使用思源黑体或阿里巴巴普惠体；英文标题可使用 Georgia，与展示站品牌字形成呼应。

### 固定结构

1. 顶部 8%：章节导航，当前章节使用铜橙色。
2. 左上：页标题和一句结论。
3. 中部：一张主视觉，最多两张辅助图。
4. 底部或右侧：2–4 个证据点，不堆叠长段落。
5. 页码放右下角，保持低对比度。

### 截图处理

- 统一裁成 16:9 或 4:3 内容框，不拉伸。
- 保留真实界面，不套厚重电脑模型。
- 用 1 px 浅灰描边和轻微阴影区分背景。
- 标注只指出真实功能，不增加界面中不存在的能力。
- 同一页最多出现 3 个标注点。

### 图表处理

- 产品流程使用横向流程图。
- 技术架构使用分层图。
- 任务调度使用时序或数据流图。
- 路线图使用四阶段横轴。
- 没有可验证数据时，不制作市场规模、准确率、性能提升等柱状图。

## 7. 制作最终 PPT 前需要确认

1. **使用场景**：比赛答辩、课程汇报、项目路演还是内部技术汇报？
2. **时长**：5 分钟、8 分钟还是 10 分钟以上？
3. **署名信息**：正式产品名、团队名、学校、成员和指导教师。
4. **实施状态**：后端三层架构中哪些已有代码，哪些正在开发，哪些只是路线图？
5. **真实指标**：是否有分类准确率、审查耗时、Token 压缩率、任务成功率或用户测试结果？
6. **是否保留商业章节**：若需要市场分析、竞品、商业模式或团队页，需要提供可靠资料和引用来源。

## 8. 当前推荐

先确认第 7 节中的使用场景、时长和技术实施状态。若没有真实业务指标，最终 PPT 应以“完整产品流程 + 三项技术创新 + 可验证演示”为主，不强行模仿参考 PPT 中的市场规模和商业成果页。
